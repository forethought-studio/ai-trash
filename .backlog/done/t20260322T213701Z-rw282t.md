    Here's exactly what happened, in the order I discovered it:

    1. The first fix attempt made CLISymlink.runOsascript() use NSAppleScript with do shell script rm -f /usr/local/bin/steadymount (no explicit path for rm).
    2. When I ran steadymount uninstall-cli to verify, it failed. The diagnostic log showed:
    sh: /usr/local/bin/rm: Operation not permitted
    3. That error path — /usr/local/bin/rm — was the clue. The do shell script shell inherits the user's PATH, which has /usr/local/bin before /bin. When rm was looked
     up, the shell found /usr/local/bin/rm first instead of /bin/rm.
    4. Running ls /usr/local/bin/rm confirmed the file exists — that's the ai-trash tool (root-owned). From the earlier ls output:
    -rwxr-xr-x   1 root  wheel   9966 Mar 22 20:11 ai-trash
    4. You've installed something that shadows rm with a trash-based wrapper.
    5. When do shell script (from a sandboxed process) tries to execute the root-owned /usr/local/bin/rm, it fails with Operation not permitted because the sandboxed
     process can't exec a root-owned binary in that context.
